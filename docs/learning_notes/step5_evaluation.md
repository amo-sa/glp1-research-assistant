# Step 5 — Evaluation harness (Ragas)

## What was built
- `data/golden_test_set.json`: 15 hand-curated (question, reference_answer,
  relevant_pmids, category) triples across four categories:
    - factual (4): directly answerable from specific chunks
    - comparative (4): requires synthesizing across multiple chunks
    - partial (3): corpus has some but not complete evidence
    - out_of_scope (4): correct answer is explicit refusal
  Reference answers verified manually against actual cached PubMed XML
  before use -- not generated by LLM. This is what makes it "golden."

- `src/run_pipeline_for_eval.py`: runs the full multi-agent RAG pipeline
  (research_graph.py) on all golden set questions and saves outputs to
  data/processed/pipeline_outputs.json. Intentionally imports nothing from
  Ragas -- completely separate dependency set.

- `src/evaluate.py`: reads pipeline_outputs.json and scores each result
  using Ragas across four metrics. Intentionally imports nothing from the
  pipeline (no langgraph, no chromadb, no sentence-transformers). Exits
  with code 0 if all metrics pass thresholds, code 1 if any fail -- the
  CI gate in Step 8 reads this exit code.

- Two-script architecture was a deliberate design fix: originally one
  combined evaluate.py tried to import both the pipeline and Ragas, causing
  a langchain-core version conflict (pipeline needed 0.4.x, Ragas 0.3.9
  needed 0.3.x). Decoupling into two scripts with a JSON file as the
  interface eliminated the conflict entirely.

## Baseline eval results (first full run, 15 questions)
  faithfulness:       0.722   PASS  (threshold: 0.70)
  answer_relevancy:   0.427   FAIL  (threshold: 0.70)
  context_precision:  0.506   FAIL  (threshold: 0.60)
  context_recall:     0.400   FAIL  (threshold: 0.60)

## What the per-question results revealed

### Retrieval failures (driving low context precision + recall)
- Q1 (semaglutide weight loss): needed PMID 42225300 RESULTS but retrieved
  42248359 OBJECTIVE, 42217114 CONCLUSION, 42297751 DISCUSSION -- never
  surfaced the right section despite the right paper being in the corpus.
- Q2 (lean mass): retrieved 42321502 INTRODUCTION but needed its RESULTS
  section (which contains the specific -1.74kg absolute lean mass number).
- Root cause: embedding model is better at matching questions to
  BACKGROUND/INTRODUCTION text (which shares vocabulary with how questions
  are phrased) than to RESULTS text (which contains statistics and clinical
  terminology). Known problem called the query-document semantic gap.

### Redundancy problem confirmed at scale
- Q3: PMID 42225300 retrieved 3 times out of 5 slots
- Q8: PMID 42209204 retrieved 3 times out of 5 slots
- The redundancy problem observed in Step 3/4 testing shows up consistently
  across the full eval set, not just in individual test cases.

### Answer relevancy artificially low (metric design issue)
- Every out-of-scope question scores near-zero on answer relevancy because
  Ragas generates a question from "The provided excerpts do not contain
  sufficient information..." which is semantically unrelated to the original
  question. A faithful refusal is indistinguishable from an irrelevant
  answer under this metric. Fix: filter out-of-scope questions from the
  relevancy metric, or use a separate metric for refusal quality.

### Real hallucination caught (Q15, SGLT2 inhibitors)
- System correctly stated SGLT2 mechanism wasn't in the excerpts, then
  immediately answered from parametric memory anyway ("based on general
  knowledge, SGLT2 inhibitors work by blocking...") before partially
  self-correcting. The critique agent approved this answer -- it either
  missed the parametric memory leak or judged the self-correction sufficient.
  Clearest real-world faithfulness failure in the eval set.

### Reference answer gaps (Q6, Q11)
- Q6 (GI side effects): generated answer found specific GI symptoms from
  additional PMIDs not in our reference answer -- system arguably produced
  a better answer than the reference. Penalized by eval because it doesn't
  match the incomplete reference. Real limitation of reference-based eval:
  an answer better than your reference gets a low score.
- Q11 (mental health): reference said "not sufficient information" but
  system found PMID 42252120 (psychiatric comorbidity) and 42223191
  (eating disorders) -- genuinely relevant content. Golden set entry should
  be updated to reflect what's actually in the corpus.

### Subtle faithfulness cases
- Q2: "unfavorable effects on lean mass" loses the important clinical
  nuance that lean mass proportion improved while absolute lean mass
  decreased -- a simplification that changes clinical meaning.
- Q14 (bariatric surgery): correctly declined direct comparison but added
  "used as adjuncts to, not replacements for, bariatric surgery" which
  may be an inference not directly stated in retrieved chunks.

## Key concepts

### Four Ragas metrics
- Faithfulness: extracts individual claims from response, checks each
  against retrieved_contexts via judge LLM, score = supported/total.
  Does NOT need a reference answer.
- Answer Relevancy: generates N questions from the response, embeds them
  and the original question, score = average cosine similarity. Does NOT
  need a reference answer. Penalizes faithful refusals -- known limitation.
- Context Precision: for each retrieved chunk, asks judge LLM "is this
  relevant given the question and reference?", position-weighted so
  earlier relevant chunks score higher. NEEDS reference answer.
- Context Recall: extracts claims from reference answer, checks each
  against retrieved chunks, score = supported reference claims / total.
  NEEDS reference answer.

### LLM-as-a-judge
- Ragas uses an LLM internally to compute metrics -- not string matching
  or BLEU/ROUGE. Handles paraphrases correctly, captures semantic meaning.
  Tradeoff: judge LLM can be wrong or biased, scores are estimates not
  ground truth. But estimates are consistent and reproducible, which is
  what matters for catching regressions in CI.
- Same-model judging: we used Llama 4 Scout as both generator and judge.
  Known limitation -- a model may be biased toward its own outputs. Fix:
  use a different/stronger model as judge. DeepSeek R1 is available on
  Jetstream and would be a better judge for faithfulness (reasoning model
  explicitly thinks through "is this claim supported?" before deciding).

### SingleTurnSample and EvaluationDataset
- Named container classes -- hold data in the shape Ragas expects.
  SingleTurnSample: one (user_input, response, retrieved_contexts,
  reference) tuple. EvaluationDataset: a typed wrapper around a list of
  samples that validates structure and handles batching.

### Metric injection pattern
- Ragas metric objects (faithfulness, answer_relevancy, etc.) are
  pre-instantiated module-level objects with llm and embeddings attributes.
  Inject our judge: metric.llm = judge_llm. hasattr() check before setting
  embeddings because not all metrics use them (only answer_relevancy does).

## What broke / had to fix

### Dependency conflicts (the biggest challenge in this step)
- Ragas 0.4.3: imports langchain_community.chat_models.vertexai which was
  removed in langchain-community 0.4.x. Fixed by creating a stub file
  at the missing import path.
- Ragas 0.3.9 + langchain-core 0.4.x: compatible for Ragas but broke
  langgraph (which needs langchain-core 0.4.x module paths). Resolved by
  decoupling into two scripts so the two dependency sets never share a
  process.
- llm_factory (Ragas's recommended replacement for LangchainLLMWrapper):
  creates an InstructorLLM object that requires structured output extraction
  capabilities our Jetstream endpoint doesn't support. Reverted to
  LangchainLLMWrapper despite deprecation warning -- the suggested
  migration path doesn't work for non-OpenAI endpoints.
- results["faithfulness"] returns a list in Ragas 0.3.9, not a float --
  had to average manually with numpy.

### Golden test set corrections after manual verification
- Q1: removed PMID 42297754 from relevant_pmids (paper is about QoL, not
  weight loss as primary outcome) and updated reference answer
- Q5 (GI side effects): removed "nausea, vomiting, diarrhea" from
  reference answer -- those specific terms aren't in PMID 42201797
- Added Q4 (semaglutide MACE reduction) using PMID 42316327 which has
  specific verifiable RR statistics (0.71, 95% CI 0.60-0.84)
- Manual verification of all 8 PMIDs in relevant_pmids against actual
  cached XML before writing any eval code

## Likely interview questions tied to this step
- "How did you build your evaluation harness?" -> golden test set
  construction (human-verified reference answers, four question categories),
  Ragas metrics and what each measures, LLM-as-a-judge approach, CI gate
  via exit codes.
- "What do faithfulness, context precision, and context recall actually
  measure, and why do you need all three?" -> faithfulness = generation
  quality (are claims supported?), precision = retrieval noise (did we
  pull relevant chunks?), recall = retrieval completeness (did we get
  everything we needed?). Can't infer one from the others -- a system
  can have perfect precision with low recall (selective but incomplete)
  or perfect recall with low precision (complete but noisy).
- "What did your eval results show?" -> faithfulness passed (0.72),
  context precision (0.51) and recall (0.40) identified retrieval as the
  bottleneck, answer relevancy (0.43) was artificially low due to faithful
  refusals being penalized. Specific actionable finding: embedding model
  prefers BACKGROUND sections over RESULTS sections, causing the
  query-document semantic gap.
- "What's wrong with answer relevancy as a metric for your system?" ->
  penalizes correct refusals on out-of-scope questions equally to irrelevant
  answers. A faithful "I don't know" scores near-zero. Fix: filter by
  question category before computing this metric, or use a refusal-quality
  metric instead for out-of-scope questions.
- "How would you improve these scores?" -> section-preference reranking
  (boost RESULTS/CONCLUSIONS in top-k), max-chunks-per-source limit to
  fix redundancy, query rewriting to bridge the query-document semantic gap,
  hybrid search (semantic + BM25) to capture exact term matches.
- "What's the most important thing you caught with this eval harness?"
  -> Q15's SGLT2 hallucination: system correctly said the answer wasn't
  in the corpus then answered from parametric memory anyway, and the
  critique agent approved it. Manual inspection of 15 answers found this;
  the faithfulness metric score of 0.72 (not 1.0) quantifies it across
  the full set.

## Known gaps / things not yet handled
- Out-of-scope questions should be evaluated differently -- refusal quality
  is a different dimension from answer relevancy. Current setup penalizes
  correct refusals. Fix: compute answer_relevancy only on in-scope
  questions, use a separate binary metric (did the system correctly refuse?)
  for out-of-scope.
- Same-model judging: Llama 4 Scout judging its own outputs. DeepSeek R1
  available on Jetstream would be a stronger, unbiased judge. Next step:
  re-run evaluate.py with R1 as judge (pipeline outputs already cached,
  no re-run needed) and compare scores.
- Q11 golden set entry needs updating: reference says "not sufficient
  information" but corpus actually contains relevant content about
  psychiatric comorbidity and eating disorders. Reference answer should
  be updated to reflect this.
- Thresholds (0.70/0.60) are starting points, not empirically justified.
  Should be tuned after seeing scores across multiple prompt variants and
  retrieval configurations.
- pipeline_outputs.json is committed but represents one specific run --
  if the pipeline changes, it needs to be regenerated before running eval.
  CI should always regenerate rather than using cached outputs.

## Iterative improvements after baseline

### Run 2: retrieval deduplication fix
- Changed search_agent to fetch n_results=15 from Chroma instead of TOP_K=5,
  then apply MAX_PER_SOURCE=2 limit while walking results in distance order
  until TOP_K diverse chunks are collected.
- This correctly addresses the redundancy problem: redundant slots are now
  filled by the next best diverse candidate from the larger pool, rather than
  just being dropped (which was the naive first attempt at the fix).
- Result: context precision +0.010, context recall +0.033. Small improvement
  because the root cause for the worst retrieval failures (Q1) is the
  query-document semantic gap, not redundancy -- the right paper simply
  doesn't rank in the top 15 candidates for that question regardless of
  deduplication.
- Answer relevancy +0.146 from filtering out-of-scope questions from that
  metric -- removes the artificial penalty for correct refusals.

### Run 3: golden set reference answer corrections
- Q6 (GI side effects): expanded reference answer to include specific
  symptom names (nausea, bilious vomiting, constipation, diarrhea) found
  in PMIDs 42243968 and 42247124 -- the system was finding this evidence
  but getting penalized for "going beyond" an incomplete reference.
- Q11 (mental health): updated from "not sufficient information" to a
  substantive reference answer reflecting content actually in the corpus
  (psychiatric comorbidity data from PMID 42252120, eating disorder
  relationship from PMID 42223191). System was correctly finding and citing
  this evidence; the reference answer was wrong, not the system.
- Result: faithfulness 0.739 (highest across all runs), answer relevancy
  0.615 (continued improvement from 0.427 baseline).

### Final scores across runs (interview-ready summary)
  Baseline:           faithfulness 0.722, relevancy 0.427, precision 0.506, recall 0.400
  After dedup fix:    faithfulness 0.692, relevancy 0.573, precision 0.516, recall 0.433
  After golden fix:   faithfulness 0.739, relevancy 0.615, precision 0.480, recall 0.411

  Key improvement: answer relevancy +0.188 from two targeted fixes.
  Remaining gap: context precision/recall stuck around 0.48/0.41.
  Root cause identified: query-document semantic gap -- questions are phrased
  differently from how research findings are written, causing embedding model
  to prefer BACKGROUND sections over RESULTS sections. Fix requires query
  rewriting (deferred to polish pass).

### Lesson: eval harness drove all three improvements
  Without the per-question breakdown in eval_results.json, we would not have
  known: (1) which questions had retrieval failures vs. generation failures,
  (2) that Q6's reference answer was incomplete, (3) that Q11's reference
  answer was wrong. The eval harness didn't just measure quality -- it told
  us exactly what to fix and in which direction.
